# CLAUDE.md — working notes for fpv-sim-mcp

MCP server exposing the [fpv-sim](https://github.com/wasomma/fpv-sim)
simulation as deterministic, read-only tools, built on a headless
TypeScript port of that repo's `index.html` sim core. Start with
README.md and DESIGN_NOTES.md; version history in CHANGELOG.md.

## The one invariant that matters

**The browser file is the specification; this engine is its verified
twin.** Golden-master fixtures (`test/fixtures/golden-seeds.json`) are
generated by running the ORIGINAL upstream source headless
(`scripts/generate-goldens.mjs`, requires a `../fpv-sim` sibling
checkout) and the engine must reproduce them exactly — string-equal
events, float-equal CEPs, no epsilons. Policy:

- **A failing golden test means the engine is wrong** — never "fix" a
  fixture to make a test pass.
- Fixtures are regenerated only when upstream changes behavior, together
  with the new upstream commit recorded in `docs/upstream/SNAPSHOT.md`
  and a CHANGELOG entry.
- The weekly `upstream-drift` workflow re-derives fixtures from live
  upstream main and fails on divergence: step "Engine reproduces the
  committed fixtures" failing = engine regression; step "Committed
  fixtures match live upstream" failing = upstream changed (stale spec).
- Preserve RNG draw order and floating-point expression order in any
  engine edit — they ARE the determinism contract (`src/engine/rng.ts`).

## Layout & commands

- `src/engine/` — the port; `src/server/` — MCP tools (stdio `index.ts`,
  Streamable HTTP `http.ts`, shared `build.ts`); `src/server/params.ts`
  is the single tunable-parameter table that generates BOTH input
  validation and `get_config_schema` (and fpv-sim's PARAMETERS.md).
- `npm test` — build + 17 tests (golden parity, fix math, aggregation).
  Must pass on Node 20, the stated minimum — the test glob is deliberately
  unquoted so the shell expands it there.
- `npm run demo` — exercises the server through a real MCP stdio client.
- `npm run goldens` — regenerate fixtures (needs `../fpv-sim`).
- `deploy/` — VPS runbook for the hosted demo (wasomma-fpv.duckdns.org);
  deploys are manual, so engine/server changes don't reach it until
  someone follows DEPLOY.md.

## Conventions

- CHANGELOG.md (Keep a Changelog + semver) and `package.json` `version`
  move together.
- Consumers to keep in mind: fpv-sim's `scripts/monte-carlo-study.mjs`
  and `scripts/generate-parameters-doc.mjs` import this package's built
  `dist/` from a sibling checkout — build breaks there usually mean a
  build wasn't run here.
- Scheduled workflows pause after ~60 days of repo inactivity — GitHub
  emails a one-click re-enable.
